Brickles · Design Spec

Worlds & the Level Select screen

A written proposal for two things: (1) formalising the soft-pastel + dark-ink-outline visual language and mapping it onto two worlds, and (2) a level-select screen with a real unlock + save system. Concrete hex throughout — the background is code, not art, so every value here is build-ready.

Scope · spec + mockup, not implementation Canvas · 1080 × 1320 portrait Ink · #34215a Levels · 10 → 2 worlds
01

The pastel-and-ink language

Three rules make everything look like it belongs in the same sticker book.

Rule 1 — One ink

Every shape is outlined in the same deep-grape ink — #34215a. Never black, never per-colour outlines. This single line weight + colour is what fuses the palette together.

Rule 2 — Pastel fill, three tones

Each hue ships as a trio: a fill, a darker shade (text / bottom edge), and a light (sheen / particles). Straight from RB in levels.ts.

Rule 3 — Chunky drop, no blur

The “sticker” depth is a hard offset shadow in ink (e.g. 0 6px 0 #34215a) plus a top sheen — not a soft blur. Corners are generously rounded.

Data-model change

Two small additions to levels.ts. Add an explicit world field to Level (explicit beats deriving from index — it survives reordering and lets a world span any range later), plus a WORLDS metadata array that owns the per-world theme so render.ts can read sky stops without hard-coding them.

// levels.ts — additions
export interface Level {
  name: string;
  grid: string[];
  world: number;        // NEW · 1-based world id
}

export interface World {
  id:   number;
  name: string;        // "Meadow & Sky"
  mood: string;        // "daytime" | "candy-night" — label only
  sky:  [string,string,string]; // gradient stops, top→bottom
  twinkles: number;    // decor density override (26 today)
}

export const WORLDS: World[] = [
  { id:1, name:'Meadow & Sky',     mood:'daytime',
    sky:['#9ed8ff','#c5ecff','#eafff4'], twinkles:18 },
  { id:2, name:'Starlight & Sugar', mood:'candy-night',
    sky:['#4a3b8f','#8a5fc0','#d98fd0'], twinkles:40 },
];

// each LEVELS entry gains:  world: 1  (×5)  then  world: 2  (×5)

Backwards-compatible: if world is ever missing, fall back to Math.floor(idx / 5) + 1. drawBackground() gains one argument — the active World — and reads world.sky / world.twinkles instead of today's constants.

03

Unlock rule & where progress lives

There's no save system yet — only brickles_hi. Here's the smallest one that makes a level-select meaningful.

The unlock rule

  • Level 1 is always unlocked.
  • Beat level N → unlock level N+1. Linear; the existing levelClear() already fires at the exact right moment.
  • Cleared levels stay cleared and are replayable — re-playing can only raise your best score, never re-lock anything.
  • Worlds gate by their first level. World 2 “opens” simply because beating level 5 unlocks level 6 — no separate world lock needed.

A level is therefore in exactly one of three states, derived — not stored separately:

Completed — best[idx] exists
Unlocked — idx ≤ maxUnlocked
Locked — idx > maxUnlocked

Where it persists — localStorage

One new key, sitting beside the existing brickles_hi. Stores the furthest unlock and a per-level best so completed tiles can show a number.

// key: "brickles_progress"
interface Progress {
  maxUnlocked: number;            // 0-based, highest playable idx
  best: Record<number,number>;     // idx → best score
}

// on levelClear(idx, score):
p.maxUnlocked = Math.max(p.maxUnlocked, idx + 1);
p.best[idx]   = Math.max(p.best[idx] ?? 0, score);
localStorage.setItem('brickles_progress', JSON.stringify(p));

Default when absent: { maxUnlocked: 0, best: {} } — only level 1 open. Capped at LEVELS.length - 1. No accounts, no server; clears with site data, same as the high score.

How it fits the existing flow

One new state — 'levelselect' — joins the seven in game.ts. It's reachable from the screens that already exist, and every tile tap routes through the buildLevel(idx) you already have.

title →Levels btn levelselect →tap tile serve → play → levelclear ↘Level Map levelselect

Entry points: a “Choose Level” button on the title overlay (next to Start, which now means “continue at your furthest level”); a “Level Map” button on the levelclear overlay (beside Next) and on gameover. Selecting a tile calls buildLevel(idx); state = 'serve'; hideAll(); — identical to nextLevel() but with a chosen index.

✦

Recommendation — build it as an HTML overlay, not a canvas state

Pause, game-over and title are already .overlay divs toggled by show()/hideAll(). A level grid is fundamentally a set of buttons with state — making it a 12th overlay-levelselect div reuses that exact machinery, gets keyboard/tap focus and scrolling for free, and keeps the render loop untouched. A canvas-drawn grid would mean hand-rolling hit-testing, focus and scroll for zero visual gain. Match the existing pattern.

Layout notes for build

  • Bands — one per world, stacked. Header is a chunky pill (world number disc in the world's signature hue + name + “x / 5 cleared”) followed by a dashed rule.
  • Tiles — 5 across, ~168px, joined by a dashed ink trail behind them for the board-game “map” read. On a phone the screen is portrait so two rows of five fit without scrolling; if more worlds arrive later, the overlay scrolls vertically.
  • Top bar — back chevron (to title) left, “Choose a Level” centre, “6 / 10 ★” progress right.
  • No new art — every pixel here is CSS/canvas primitives + the existing ink + RB palette + Fredoka. Nothing needs an illustrator.
